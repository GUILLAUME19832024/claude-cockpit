🌿 Git — base — snapshots & filet de sécurité
CommandeCe que ça faitAnalogie GUIPowerShell réelClaude CodeRisque
git status
État du dépôt
 Fichiers modifiés, ajoutés, supprimés. Ton premier réflexe avant/après Claude. Panneau Source Control identique natif Sûr
git diff
Détail des changements
 Exact détail de chaque ligne. Vert = ajouté, rouge = supprimé. Comparer deux versions identique natif Sûr
git log --oneline
Historique compact
 Liste tous les commits avec ID court + message. Historique de versions identique natif Sûr
git add .; git commit -m "..."
Checkpoint — AVANT session Claude
 Sauvegarde l'état actuel. PS : && → ; Enregistrer version nommée identique avec ; recommandé Sûr
git push
Envoyer vers GitHub
 Pousse les commits locaux vers le dépôt distant. Bouton Sync/Push identique natif Sûr
git checkout -b [branche]
Créer + basculer sur une branche
 Crée la branche et s'y place. Bonne pratique avant toute session Claude longue. Créer nouvelle branche identique recommandé Sûr
git restore [fichier]
Annuler un fichier (chirurgical)
 Remet UN fichier à l'état du dernier commit. Annuler modifications fichier identique précaution Attention
git stash
Mettre de côté temporairement
 Sauvegarde les changements sans committer. Récupérer avec git stash pop. Mettre en attente identique précaution Attention
git checkout .
Rollback TOTAL — bouton panique
 Annule TOUTES les modifications non committées. Reset total git restore . dernier recours DANGER

# CLAUDE.md — Pack IA Manager ## Objectif MVP pedagogique pour apprendre une architecture full-stack : Spring Boot, PostgreSQL, Docker, Angular. ## Stack - Backend : Java / Spring Boot - Base : PostgreSQL - Migrations : Flyway - Front : Angular - Tests API : Postman - Versionnement : GitHub ## Regles de travail - Toujours expliquer avant de modifier. - Proposer un plan avant toute modification importante. - Ne jamais supprimer de fichier sans confirmation explicite. - Ne jamais lire ni exposer les valeurs de .env. - Toujours rappeler les commandes de test. - Toujours resumer les fichiers modifies. ## Workflow Git - Avant modification : git status puis checkpoint. - Apres modification : git diff puis test puis commit. ## References - @README.md - @docs/architecture.md - @docs/security.md ## Regle de taille - Garder ce fichier synthetique (< 200 lignes cible formation). - Deporter le detail dans des fichiers @references.
🛠️ Skills — procédures réutilisables
Définition — Une skill est une procédure réutilisable que Claude Code peut appeler pour une tâche précise. Elle vit dans un dossier avec un SKILL.md + éventuellement des fichiers annexes.
Skillaudit-git
Vérifier status, diff, secrets, risques avant/après session. Read-only.
Skillexplique-code
Expliquer un fichier sans modification. Pédagogique, lien avec la stack.
Skillreview-spring
Vérifier controller / service / repository / DTO. Signale incohérences d'architecture.
Skillcheck-secrets
Détecter .env, passwords, tokens dans les fichiers ou commits.
Skillprepare-commit
Proposer message de commit et résumé des fichiers modifiés.
Mini-exemple — SKILL.md
# SKILL.md — audit-git ## Objectif Analyser l'etat Git avant/apres session Claude Code. ## Instructions 1. Lire `git status`. 2. Lire `git diff`. 3. Identifier fichiers sensibles (.env, .pem, clefs). 4. Dire si commit possible. 5. Ne jamais modifier le code.
🤝 Subagents — assistants spécialisés
Définition — Un subagent est un assistant spécialisé appelé pour une mission précise, avec ses propres instructions et parfois ses propres outils. Commande : /agents
Agentcode-reviewer
Relit le code, signale risques. Read-only.
Agentspring-architect
Contrôle l'architecture Spring Boot (layers, DI, DTO, exceptions).
Agentsecurity-guard
Cherche secrets, suppressions dangereuses, commandes risquées.
Agentlearning-tutor
Explique sans modifier. Répond avec pédagogie et fait les liens.
Agenttest-planner
Propose tests Postman / unitaires sans implémenter directement.
Règle — Au début, privilégier des agents read-only. Ne pas multiplier les agents qui modifient le code.
🪝 Hooks — déclenchement automatique d'actions Avancé / prudence
Définition — Un hook déclenche automatiquement une action à certains moments de Claude Code. Puissant mais risqué : un hook mal écrit peut bloquer ou polluer chaque session.
Points de déclenchement
HookPreToolUse
Avant qu'un outil soit utilisé (lecture, écriture, bash…).
HookPostToolUse
Après un outil réussi. Logging, audit.
HookUserPromptSubmit
Avant traitement d'un prompt utilisateur.
HookPreCompact
Avant compactage du contexte.
HookSessionEnd
Fin de session. Utile pour résumé ou export automatique.
Cas d'usage utiles
CasBloquer lecture de .env
PreToolUse qui refuse tout accès à .env*.
CasBloquer suppressions
PreToolUse qui refuse rm, Remove-Item, Delete.
CasRappel git diff
PostToolUse qui affiche un git diff après chaque édition.
CasRésumé de session
SessionEnd qui produit un résumé des fichiers touchés.
Règle — Ne pas activer de hook automatique tant que le workflow Git n'est pas maîtrisé.

🧭 Comprendre Claude Code — leviers et portées — mecanismes fonctionnels puis niveaux d'heritage V3.8.4 · lecture avancée
Point clé : les leviers fonctionnels ne sont pas des niveaux hierarchiques. Ce sont des mecanismes complementaires : memoire, expertise, controle et delegation. La hierarchie existe ailleurs : dans les niveaux de portee des dossiers et fichiers charges par Claude Code.
OFFICIEL ANTHROPICVOCABULAIRE PEDAGOGIQUE
Leviers fonctionnels — pas une hierarchie
Ces quatre leviers se combinent. Ils ne signifient pas “niveau 1, niveau 2, niveau 3, niveau 4” dans les dossiers.

Memoire & consignes persistantesCLAUDE.md

Ce que Claude doit savoir a chaque session : architecture, conventions, commandes, pieges connus, fichiers de reference.

Usage : stabiliser le contexte projet.

/init

Competences reutilisablesSkills · SKILL.md

Expertises specialisees activables au bon moment : revue de code, tests, securite, architecture, documentation.

Usage : capitaliser des pratiques d'equipe.

/skills

Garde-fous deterministesHooks · permissions

Controles automatiques qui ne dependent pas du raisonnement du modele : blocage, lint, tests, journalisation.

Usage : securiser les actions sensibles.

/hooks

Delegation specialiseeSubagents

Sous-agents dedies a des taches precises, avec leur role, leur contexte et leurs outils autorises.

Usage : separer les roles et limiter la surcharge contextuelle.

/agents
Niveaux de portee & heritage de contexte
Machine / utilisateur └─ ~/.claude/ ├─ CLAUDE.md → preferences et regles personnelles └─ skills/ → competences personnelles reutilisables Espace parent / workspace └─ CLAUDE.md → regles communes a plusieurs projets ou au monorepo Projet └─ ./mon-projet/ ├─ CLAUDE.md → memoire projet partagee ├─ .claude/settings.json → permissions et configuration projet ├─ .claude/skills/ → skills partagees equipe ├─ .claude/commands/ → commandes slash personnalisees └─ .claude/agents/ → subagents projet Module / sous-dossier └─ ./mon-projet/frontend/ └─ CLAUDE.md → regles specifiques au module frontend

🛡️ Routine quotidienne Claude Code — apprendre sans casser

A. Avant session

  1. pwd — verifier le dossier.
  2. git status — voir l'etat.
  3. git diff — comprendre les changements.
  4. git add .; git commit -m "checkpoint avant claude"
  5. claude --permission-mode plan

B. Pendant session

  1. /plan — clarifier avant action.
  2. /status — verifier modele, compte, contexte.
  3. /context — surveiller saturation.
  4. /compact [focus] — compacter sans perdre le fil.
  5. /diff — auditer regulierement.
  6. /rewind — revenir si Claude part trop loin.

C. Apres session

  1. /diff — verifier les modifications.
  2. git status — controler fichiers touches.
  3. Lancer tests/lint/build si disponibles.
  4. git add .
  5. git commit -m "..."
  6. git push si partage necessaire.
Regle personnelle : en apprentissage, ne jamais utiliser dangerously-skip-permissions, ne jamais accepter une modification importante sans plan, sans /diff, et sans checkpoint Git.

## Règles de travail avec Claude Code ### Avant toute modification - Toujours expliquer avant de modifier. - Proposer un plan détaillé avant toute modification structurelle. - Limiter une modification à un seul fichier sauf accord explicite. - Ne jamais supprimer de fichier sans confirmation explicite. - Ne jamais lire ni exposer les valeurs de .env, tokens ou secrets. ### Pendant la modification - Annoncer chaque fichier avant de le toucher. - S'arrêter en cas de doute plutôt que d'inventer. - Ne pas committer sans validation explicite. ### Après modification - Résumer les fichiers modifiés et leur diff principal. - Rappeler les commandes de test à lancer. - Signaler tout changement non explicitement demandé. ### Style attendu - Code lisible avant performance. - Commentaires utiles, pas décoratifs. - Solutions idiomatiques pour la stack courante.
